# PAC Phase 3.1 — Geo Intelligence Engine Implementation Plan

This implementation plan details the architecture, design, and file modifications required to build a production-ready spatial hotspot clustering and Geo Intelligence module.

---

## User Review Required

> [!IMPORTANT]
> - **Clustering Engine**: We will use PostGIS native `ST_ClusterDBSCAN` projected to EPSG:3857 (Spherical Mercator) to ensure `eps` is processed accurately in meters.
> - **Clustering Metrics**: For each hotspot, we will return:
>   - **Risk Level**: High (crimes >= 10 OR repeat offenders >= 2 OR gangs >= 1), Medium (crimes >= 5 OR repeat offenders >= 1), or Low (otherwise).
>   - **Repeat Offenders Count**: Distinct repeat offenders (`is_repeat_offender = true`) linked to crimes in the cluster.
>   - **Known Gangs Count**: Distinct `gang_name` counts of gang-affiliated criminals linked to crimes in the cluster.
>   - **Hotspot Trend**: `Emerging` (if counts in the second half of the date range are higher than the first half), `Declining` (if lower), or `Stable` (otherwise).
>   - **Suggested Patrol Window**: A rule-based window that pads the peak time by 30 minutes on both sides (e.g., peak `18:00-22:00` pads to `17:30-22:30`).
>   - **Deterministic AI Recommendation**: Rules-based tactical operational recommendations generated by a standalone recommendation engine component.

---

## Future Module Alignment

### 👤 Behaviour Profiles (Phase 3.3)
The trend analysis (`Emerging`, `Stable`, `Declining`) and spatial metrics will be utilized by the Behaviour Profile module to:
- Cross-reference known offender operating radii (`operating_radius_km`) and preferred crime types.
- Alert when an offender's home location falls within an **Emerging** hotspot.
- Track spatial displacement of offender patterns when target areas become high-risk.

### 🤖 AI Investigation Assistant (Phase 4)
The standalone `recommendation_engine.py` is designed to be fully modular:
- Currently, it generates deterministic rule-based operational recommendations.
- In Phase 4, the deterministic logic can be wrapped or replaced by an LLM (Gemini) endpoint. The LLM will consume the structured response (crime counts, offender details, trend, spatial metrics) to generate context-aware patrol briefings or respond to conversational queries like `"Explain why the Marathahalli hotspot is emerging."`

## Proposed Changes

### Database Layer

#### [NEW] [geo_repo.py](file:///c:/Users/Teja/OneDrive/Desktop/PAC/backend/app/repositories/geo_repo.py)
Implements queries using PostGIS:
1. **DBSCAN Clustering SQL**:
   ```sql
   WITH clustered_crimes AS (
       SELECT
           c.id,
           c.crime_type,
           c.occurred_at,
           c.district,
           c.police_station,
           c.latitude,
           c.longitude,
           c.geom,
           ST_ClusterDBSCAN(ST_Transform(c.geom, 3857), eps := :eps, minpoints := :min_samples) OVER () AS cluster_id
       FROM crimes c
       WHERE c.geom IS NOT NULL
         AND (:district IS NULL OR c.district = :district)
         AND (:crime_type IS NULL OR c.crime_type = :crime_type)
         AND (:start_date IS NULL OR c.occurred_at >= :start_date)
         AND (:end_date IS NULL OR c.occurred_at <= :end_date)
   )
   SELECT
       cc.cluster_id,
       COUNT(*) AS crime_count,
       ST_Y(ST_Centroid(ST_Collect(cc.geom))) AS center_latitude,
       ST_X(ST_Centroid(ST_Collect(cc.geom))) AS center_longitude,
       COALESCE(MAX(ST_Distance(geography(cc.geom), geography(ST_Centroid(ST_Collect(cc.geom))))), 0.0) AS radius_meters,
       ARRAY_AGG(cc.id) AS crime_ids
   FROM clustered_crimes cc
   WHERE cc.cluster_id IS NOT NULL
   GROUP BY cc.cluster_id
   ORDER BY crime_count DESC;
   ```
2. **Cluster Detail Query**: Fetch MO time-of-day slots, occurred timestamps, distinct repeat offenders, and distinct gang names linked to a set of `crime_ids`.

#### [NEW] [recommendation_engine.py](file:///c:/Users/Teja/OneDrive/Desktop/PAC/backend/app/services/recommendation_engine.py)
Generates deterministic tactical recommendations based on hotspot parameters:
- Inputs: `crime_type`, `peak_time`, `risk_level`, `repeat_offenders`, `known_gangs`, `crime_count`, `hotspot_trend`.

#### [NEW] [geo_service.py](file:///c:/Users/Teja/OneDrive/Desktop/PAC/backend/app/services/geo_service.py)
Responsible for:
1. Coordinating the geo repository calls.
2. For each cluster, fetching crime occurrences to perform **Trend Analysis** (comparing count of first half of time range vs second half).
3. Calculating **Suggested Patrol Window** (padding the peak slot by 30 mins).
4. Invoking `RecommendationEngine` to get tactical advice.
5. Formatting final schema responses.
